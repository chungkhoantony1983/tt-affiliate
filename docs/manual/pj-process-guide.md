PJ（プロジェクト）管理プロセスと PM デリバリー成果物を定義する。設計プロセス（L1〜L5）は `design-process-guide.md` に委任し、本書は **G0 PJ 管理** と **PM 成果物の生成・管理** に特化する。

| ガイド | scope |

| **本ファイル (pj-process-guide)** | G0 PJ管理 + `docs/deliverables/` 配下成果物 + PM artifacts生成プロセス |

## §3: PJプロセス（フェーズ別）

### G0: PJ立ち上げ

> **目的**: PJの枠組みを確定し、設計開始の前提条件を整える。

1. **PJ計画書作成** (PJ-001): 契約書・RFP・業務ソースから目的・scope・体制・制約を定義
2. **WBS初版作成** (PJ-002): DPG成果物マップ（19件）をタスク分解の入力として、ストリーム×フェーズで分解
3. **概要見積作成** (PJ-007 概要、±25%): WBS 初版 + L1 `strategy.md` + L2 `capabilities.md` (Cap 一覧) + L2 `crc.md` (CR 概要) + ロール別単価 + DPG §11 PJ規模別スケーラビリティから工数算出。**L3 詳細・L4 横串は不要** (詳細見積は G1 末、§4 §見積の3段階 参照)
4. **リスク初期評価** (PJ-006): PJ計画書の前提条件・制約からリスクを洗い出し
5. **課題管理表初期化** (PJ-005): GitHub Issues との同期ルールを確定

**G0完了条件**: PJ-001, PJ-002, PJ-007 (概要見積 ±25%) が PO 承認済み

### G1: 設計（DPG委任）

> **目的**: DPG §5 設計プロセスに従い、L1〜L3 の設計成果物を作成する。PPGとしては進捗管理のみ。

1. **DPG §5 に委任**: L1 Strategy → L2 Domain → L3 Architecture の順で設計
2. **進捗報告** (PJ-004): 週次でWBS進捗率を更新
3. **議事録** (PJ-003): PO対話・レビュー結果を記録
4. **テスト計画書作成** (PJ-009): DPG §7 テスト規範を入力として、G1末までに作成
5. **詳細見積作成** (PJ-007 詳細、±10%): L3 全件完了 + L4 横串9観点完成後、WBS 詳細 (Component 単位 + test 工数分解) で見積再算出。Stage{N+1} の estimate file として追記 (§4 §見積の3段階 参照)

**G1完了条件**: DPG CP-3（構造設計承認、DPG §9参照）通過 + PJ-009 承認済み + PJ-007 (詳細見積 ±10%) 承認済み

### G2: 実装・検証

> **目的**: L4 実装設計 → L5 本番の順で実装・検証する（画面モックは L2 で体験検証済み、L5 で view-model seam 越しに昇格）。

1. **DPG §5 に委任**: L4 → L5
2. **進捗報告** (PJ-004): 週次更新
3. **課題管理** (PJ-005): 実装中に発見された課題を記録・追跡
4. **リスク更新** (PJ-006): 実装リスクの顕在化を監視

**G2完了条件**: DPG CP-4（実装設計承認）通過 + テスト合格

### G3: リリース・運用

> **目的**: 本番リリースと初期運用を管理する。

1. **Release Docs Freeze**: リリース前に全成果物の版を確定
2. **リリース判定**: テスト結果・残課題・リスクを総合判断
3. **運用引き継ぎ**: 運用手順書（`docs/operations/`）の確認

**Freeze手順**:
1. `docs/deliverables/` 配下の全ファイルを最終レビュー
2. GitHub Release タグを作成
3. 外部提出用ファイルを `deliverables/` から export

### G4: PJ完了

> **目的**: PJを正式に完了し、教訓を記録する。

1. **PJ完了報告書作成** (PJ-099): 成果サマリ・KPI達成度・残課題・教訓
2. **教訓の記録**: `learn()` で Memory に永続化
3. **アーカイブ**: `docs/deliverables/` を最終状態で保全

## §4: 見積・WBS

### WBSストリーム定義

|---|---|---|
| **PM** | PJ管理・進捗報告・会議 | PPG §3 |
| **設計** | L1〜L3 設計成果物作成 | DPG §5 L1/L2/L3 |
| **実装** | L4 実装設計 + L2 画面モック昇格 | DPG §5 L4 |
| **テスト** | unit/integration/e2e/NFT | DPG §7 テスト規範 |
| **リリース** | L5 本番 + 運用引き継ぎ | DPG §5 L5 |
| **インフラ** | 環境構築・CI/CD | DG |

### ロール別単価テンプレート

| ロール | 想定スキル | 単価目安（月額） |
|---|---|---|
| PM | PJ管理・顧客折衝 | PJ固有で定義 |
| アーキテクト | L2/L3設計・Cap境界判断 | PJ固有で定義 |
| エンジニア | L4実装 + 画面モック構築 | PJ固有で定義 |
| QA | テスト設計・実行 | PJ固有で定義 |

> 単価はPJ固有。テンプレート（`docs/manual/templates/estimate.template.md`）に従い記入。

### 見積の3段階 (精度 × 必要 Layer 完了度 × WBS 詳細度 × DIC + Running 費用 declared 詳細度)

各段階で出力する見積は、精度・timing だけでなく **「どの設計 Layer まで完了している必要があるか」** と **「WBS をどの粒度まで分解している必要があるか」** と **「開発インフラ費 (DIC) + Running 費用 をどの詳細度で declared するか」** が決まる (Issue #569、DR `docs/decision-records/569/estimate-rules-declared.md` + Issue #574、DR `docs/decision-records/574/dic-running-split.md` ← DR-572 supersede)。AI agent / 人間が見積出力タスクを受けた際、本表を参照して必要 Input が揃っているかを確認すること。

| 段階 | 精度 | timing | 必要 Layer 完了度 | 必要 WBS 詳細度 | DIC + Running 費用 declared 詳細度 | 根拠 |
|---|---|---|---|---|---|---|
| **概算** | ±50% | 提案時 (受注前) | — (該当なし、業務ソース概観のみ) | マイルストーン level (M-0〜M-5、WBS パッケージ未分解) | DIC + Running **項目一覧のみ** (金額 TBD) | DPG §11 PJ規模別 + 類似PJ実績 |
| **概要** | ±25% | G0 完了時 | L1 (`strategy.md`) + L2 (`crc.md` + `capabilities.md` Cap 一覧 + `capability-landscape.d2`) | WBS 初版 (5 WBS パッケージ + ストリーム×フェーズ + Cap 単位タスク粒度) | DIC + Running 各項目の **業界一般値** で declared + 発注者検証 marker | WBS 初版 + DPG §2 成果物19件マップ + L2 Cap 数 + CR 数 |
| **詳細** | ±10% | G1 末 完了時 | L1 + L2 全件 (+ `bpm/{scenario}.d2`) + L3 全件 (Cap §L3 5軸具体 + cross-Cap 横軸 + `component-diagrams/` + `api/cap/` + `ui/{screen}.md` + `bpm/{scenario}.d2` + `parnas-evaluation-matrix.md` + `threat-model.md`) + L4 横串9観点 | WBS 詳細 (Component 単位タスク粒度、per-Cap unit + cross-Cap integration + e2e + NFT 工数分解) | DIC + Running 各項目の **確定金額** (発注者 IT 部門 + ベンダー見積取得後) | WBS 詳細 + L3 完了後の実工数 + L4 横串9観点による実装工数精緻化 |

> **運用ルール**:
> - 上位段階 (詳細 → 概要 → 概算) は下位段階の前提を含む。詳細見積出力時は概要見積の前提もすべて満たすこと。
> - 「必要 Layer 完了度」の未達状態で次段階の見積を求めるのは PPG 違反。AI agent は「現状 L2 までしか完了していない → 概要見積のみ出力可能、詳細見積は L3 完了待ち」と自律判断する。
> - PO/PM 側は本表を参照し、L3 完了前に詳細見積を求めない (求める場合は段階を「概要」に修正)。
> - **開発インフラ費 (DIC) + Running 費用** の declared は §開発インフラ費 + §Running 費用 を参照 (DR-574 で旧 §継続費用 を分離)。工数見積 × 単価 (人件費) のみで見積書を完了させず、両方必ず declared する。

### §開発インフラ費 (DIC、Development Infrastructure Cost) — declared SSoT

開発工数を実行するために必要な環境費 (= **開発期間中のみ発生** する付随費用、DR-574 で declared)。本番リリース後はゼロ (環境終了)。

#### 用語明確化 (declared)

| 用語 | 定義 |
|---|---|
| **受注者経由 (Vendor-Routed)** | 受注者がベンダー契約 + 立替 + **見積に含めて発注者に請求** (見積総合計に加算) |
| **発注者直接契約 (Customer-Direct)** | 発注者が直接ベンダー契約 (**見積から除外、参考値として記載**)、契約経路の違いのみで負担は発注者 |
| **対象期間** | **G0〜G2 完了まで** (= 開発期間中)、G3 (本番リリース) 以降はゼロ |
| **申込者・利用者** | 本番運用時の業務利用者 (顧客 / エンドユーザー) は **DIC 対象外、負担ゼロ** (自明) |

#### DIC reference table (PJ 固有調整必須)

| カテゴリ | 例 | 計算根拠 (典型) | 慣習: 受注者経由 vs 発注者直接 |
|---|---|---|---|
| 開発クラウド | dev / staging 環境 (AWS / Azure / GCP) | クライアント数 / 規模依存 | 受注者経由 (開発者環境のため) |
| ソースコード管理 | GitHub Enterprise / GitLab | seat 数 × 単価 (¥3,000-5,000/seat/月) | 受注者経由 |
| CI/CD | GitHub Actions / CircleCI | チーム規模依存 (¥10K-30K/月) | 受注者経由 |
| PM ツール | Slack / Jira / Notion | seat 数 (¥1,000-3,000/seat/月) | 受注者経由 |
| IDE / 開発ツール | JetBrains 等 | 開発者数 × 単価 (¥5,000-10,000/seat/月) | 受注者経由 |
| 監視 / ログ (開発) | Datadog dev / staging | 軽量利用 (¥10K-50K/月) | 受注者経由 |
| セキュリティスキャン | Snyk / SonarQube | コードスキャン (¥10K-50K/月) | 受注者経由 |
| 開発期間中の通信 (VPN 等) | 専用線 / VPN (金融系特に) | 既存インフラ流用想定 or 別契約 | 発注者直接 (慣習、既存 IT 部門資産流用) |

> **注意**: 上記 reference table は PJ 固有調整必須。「受注者経由 vs 発注者直接」は慣習であり、発注者方針 (= 「すべて受注者経由で契約」or「一部直接」) で変動する。本 PJ で確定すること。

#### 見積総合計への加算 (DIC)

- **DIC 受注者経由 (開発期間中) 一時費用合計** = DIC 月額合計 × 開発期間 (月数)、**見積に含めて発注者請求**
- **DIC 発注者直接契約 (開発期間中) 一時費用合計** = 参考値 (見積外、発注者が直接ベンダー契約)

### §Running 費用 (Ongoing Cost、本番運用後のみ) — declared SSoT

リリース後の実際の利用時のライセンス費 / インフラ費 (= **本番運用後のみ発生** する継続費用、DR-576 で 3 区分に再構成)。PJ 完了後も継続発生。

#### 区分定義 (DR-576、受注者ビジネスモデル別)

| 区分 | 内容 | 見積反映 | 主要該当ビジネスモデル |
|---|---|:-:|---|
| **§Running 受注者提供 SaaS 利用料 (Service Revenue)** | 受注者が SaaS 提供企業の場合、発注者から継続的に受け取る月額利用料 + 従量料金 = **受注者の本番運用後の継続収益** | **見積に明示** (発注者請求の月額継続費) | SaaS 提供企業 (例: Platform) |
| **§Running 受注者内部原価 (Vendor Cost)** | 受注者が SaaS 提供のために内部で支払うベンダー費用 (Azure / Datadog / Snyk / バックアップ等) | **見積外** (受注者内部マネジメント、SaaS マージンで吸収) | 主に SaaS 提供企業 |
| **§Running 発注者直接契約 (Customer-Direct Infrastructure)** | 発注者が直接ベンダー契約する限定インフラ (専用線 / VPN / 一部監査ツール 等)、慣習依存 | **見積外、参考値として記載** | 両ビジネスモデル共通 (限定的) |

#### 受注者ビジネスモデル別 reference (declared)

##### Case A: 受注者 = SaaS 提供企業 (例: Platform、推奨フォーマット)

| カテゴリ | 例 | 区分 | 課金ロジック |
|---|---|---|---|
| Platform 共通基盤利用料 | Platform (oem base settings / design systems / external-api / Platform Operational AI / Platform Developer Kit) | Service Revenue | 月額固定 (クライアント単位) |
| BA サービス利用料 | ModuleA / EI / EP / EFB / 電子帳簿保存 / ModuleB / EL / EA | Service Revenue | 月額固定 (クライアント × BA on/off) |
| Cap 従量料金 | Cap-ADS 申込件数依存 / Cap-EID seat / API call | Service Revenue (Volume) | 件数 / seat / 呼出数 × 単価 |
| 受注者内部のクラウド原価 | Azure / AWS / GCP (本番) | 受注者内部原価 | (見積外、SaaS 提供のコスト) |
| 受注者内部の監視原価 | Datadog / NewRelic / Sentry (本番) | 受注者内部原価 | (見積外) |
| 受注者内部のセキュリティ原価 | Snyk / SonarQube / WAF | 受注者内部原価 | (見積外) |
| 受注者内部のバックアップ原価 | クラウドバックアップ | 受注者内部原価 | (見積外) |
| 通信 / VPN | 専用線 / VPN (金融系、発注者既存インフラとの接続) | 発注者直接契約 | (慣習、限定的) |

##### Case B: 受注者 = スクラッチ諸受注ベンダー (例: SIer 開発会社)

| カテゴリ | 例 | 区分 | 課金ロジック |
|---|---|---|---|
| 本番クラウド | AWS / Azure 本番 | 発注者直接契約 | (発注者が直接契約、慣習) |
| 本番監視 | Datadog 本番 | 発注者直接契約 | (同上) |
| バックアップ / DR | クラウドバックアップ | 発注者直接契約 | (同上) |
| 監査 / コンプライアンス | 監査ログ保管 / FISC / ISMS | 発注者直接契約 | (同上) |
| 通信 / VPN | 専用線 / VPN | 発注者直接契約 | (同上) |
| 受注者保守契約 (オプション) | 24/7 監視運用 / インシデント対応 | Service Revenue (限定) | 月額固定 (オプション) |

> **注意**: 上記 reference table は PJ + 受注者ビジネスモデル固有調整必須。実際の項目 / 課金区分 は受注者経営層 + 発注者 IT 部門 + ベンダーと協議して確定。

#### 用語明確化 (declared)

| 用語 | 定義 |
|---|---|
| **受注者提供 SaaS 利用料 (Service Revenue)** | 受注者が SaaS 提供企業の場合、発注者から継続的に受け取る月額利用料 + 従量料金。**受注者の継続収益**、見積に明示 |
| **受注者内部原価 (Vendor Cost)** | 受注者が SaaS 提供のために内部で支払うベンダー費用 (Azure 等)、**見積外**、SaaS マージンで吸収 |
| **発注者直接契約 (Customer-Direct)** | 発注者が直接ベンダー契約 (慣習、限定的)、**見積外**、参考値として記載 |
| **対象期間** | **G3 (本番リリース) 以降**、PJ 完了後も継続 |
| **申込者・利用者** | 本番運用時の業務利用者 (顧客 / エンドユーザー) は **Running 費用負担ゼロ** (自明)、発注者 (例: 金融機関) が全額負担 |

#### 見積書への記載必須項目 (DIC + Running 費用 共通)

1. **項目一覧** (上記 reference table + PJ 固有追加)
2. 各項目の **金額** (declared 詳細度は見積精度に連動、上記 §見積の3段階 table 参照)
3. **§Running 3 区分 (Service Revenue + 受注者内部原価 + 発注者直接契約)** の明示 (受注者ビジネスモデル別)
4. **DIC (開発期間中のみ) vs Running 費用 (本番運用後のみ)** の明示
5. **受注者請求 開発フェーズ 総合計** = 工数 + DIC 受注者経由 (一時費用)、見積に含めて発注者請求
6. **受注者請求 本番運用フェーズ 月額** = §Running 受注者提供 SaaS 利用料 (継続費)、見積に明示
7. **§Running 受注者内部原価** = 見積外 (受注者内部マネジメント、参考値として年額換算記載)
8. **§Running 発注者直接契約** = 見積外、参考値 (年額換算併記)
9. 申込者・利用者 (本番利用者) は **負担ゼロ** を明示 (自明だが見積書で明記すると誤解防止)
10. 金額が未確定の場合は TBD + 確定タイミング (例: G1 末、PO + IT 部門協議後)

#### 見積総合計の構成 (declared、DR-576 改訂)

```
[開発フェーズ、一時費用]
工数見積 (人件費 × 工数)
+ DIC 受注者経由 (開発期間中、一時費用)
+ 想定リスク (別管理、ID/工数/影響)
= 受注者請求 開発フェーズ 総合計 (税抜)
+ 消費税
= 受注者請求 開発フェーズ 総合計 (税込)

[本番運用フェーズ、月額継続費]
+ §Running 受注者提供 SaaS 利用料 (月額固定 + 従量、受注者継続収益)
= 受注者請求 本番運用フェーズ 月額 (税抜) → 税込 (発注者から受注者へ継続支払)

[参考、見積外]
- §Running 受注者内部原価 (受注者内部、SaaS マージンで吸収) = ¥XXX / 月 (受注者社内コスト)
- §Running 発注者直接契約 (発注者が直接ベンダー契約) = ¥XXX / 月 (限定項目、参考)
- DIC 発注者直接契約 (開発期間中) = ¥XXX / 月 × 期間 (限定項目、参考)
```
