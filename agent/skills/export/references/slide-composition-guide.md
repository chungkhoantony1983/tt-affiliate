# Slide Composition Guide (AI Mode)

この文書は、通常Markdownからスライド用Markdownへ変換する際の
AI構成ルールの正本。`mode=ai` 指定時にエージェントが参照する。

本テーマは **project-slide-theme.pptx** を再現する。
スライド生成時は、以下のデザイン仕様に従ってレイアウトやクラスを選択すること。

### 0-5. スライドデザイン特徴

- **フッターバー**: 各スライド下部に dk1（`#002020`）の細い帯（border-bottom）
- **h1 下線**: コンテンツスライドの h1 に accent5（`#B1DFE2`）の下線
- **背景**: lt2（`#F7FAF9`）のオフホワイト（純白ではない）
- **表紙（lead）**: lt1（`#006970`）ティール背景 + 白文字（シンプル構成、装飾なし）
- **マージン**: 左右 28px（PPTX の ~2.5mm に対応する、密度の高いレイアウト）

## 1. 全体方針

- **情報密度を最大化**: 1スライドに意味的にまとまった内容を収める
- **ページ数を最小化**: 不要な分割を避け、章単位で1〜2ページに収める
- **読み手の文脈維持**: 章の途中で改ページする場合は、前後関係が自然に繋がるようにする
- **PPTXデザインの再現**: レイアウトクラスを適切に使い分け、PPTX マスターの見た目に近づける

## 2. スライド構成ルール

### 2-1. 章とスライドの対応

- 元文書の H1/H2 見出しを **スライドタイトル**（`# タイトル`）として使う
- 1つの章（H1/H2セクション）は原則 **1〜2スライド** に収める
- 3スライド以上に分割する場合は、内容を要約して減らすことを優先する
- 「続きN」ラベルは使わない。分割が必要な場合はサブトピックで自然なタイトルを付ける

### 2-2. タイトルと本文の同居

- スライドタイトル直後に **必ず本文を開始する**
- タイトルだけのスライドは作らない（表紙・セクション区切りを除く）
- H3以下の小見出しは本文内に残し、スライド分割の基準にしない

### 2-3. 表の扱い

- 表は可能な限り1スライドに収める
- 列数が多い場合は、重要列のみに絞って補足をテキストで記載する
- 行数が多い場合（15行超）は、カテゴリごとに分割するか要約する

### 2-4. リストの扱い

- 箇条書きは1スライドあたり **6〜8項目** を目安とする
- 項目数が多い場合はグルーピングして要約する
- ネストは2階層までに抑える

### 2-5. コードブロック

- コードブロックは短いもの（10行以内）のみスライドに含める
- 長いコードは省略または要点のみ抜粋する

### 2-5a. Mermaid コードブロック

Mermaid コードブロック（`` ```mermaid ... ``` ``）は **Marp がネイティブ描画しない** ため、特別な扱いが必要。

#### 自動処理（Stage 1）

- `prepare_slides_markdown.py` (v1.4+) がスライド分割前に `mmdc` で SVG に描画し、`figures/mermaid_N.svg` として画像参照に自動置換する
- **(v1.5+)** `graph TD` / `flowchart TD` の縦方向フローチャートで行数が10行以上のものを検出し、「縦長 Mermaid フローチャート」として警告を出力する。Stage 2 で TD→LR 変換を実施すること
- **(v1.7+)** 10行以上の `graph TD` / `flowchart TD` を自動的に `LR` に変換する（Stage 1 で完了。Stage 2 での手動変換は不要）
- 描画に失敗した場合はエラー終了する（コードのままスライドに残してはならない）
- **日本語フォント**: mmdc は一時設定ファイル `_mermaid_config.json` で BIZ UDPGothic を指定し、日本語テキストの文字化けを防止する
- **テーマカラー**: ノード背景に accent6（`#E0F2F4`）、ボーダーに accent1（`#008587`）、矢印に lt1（`#006970`）を使用し、スライドテーマと統一する
- **SVG背景の透明化**: mmdc で `-b transparent` を指定し、SVG背景を透明にする。スライド背景色（lt2: `#F7FAF9`）と自然に馴染ませるため。`-b white` は使用しない
- **サイズ制御**: 図の複雑度（行数）に応じて mmdc の `-w` フラグと Marp の `w:` ヒントを自動設定する:
  - 小規模（<15行）: mmdc -w 1000 / `w:950`
  - 中規模（15-40行）: mmdc -w 1100 / `w:1000`
  - 大規模（40行以上）: mmdc -w 1200 / `w:1100`
  - ER図（80行超）/ シーケンス図（30行超）: mmdc -w 1200 / `w:1100`
- 画像参照には `center` 属性が自動付与され、図がスライド中央に配置される
- CSS テーマ側で `img[alt*="mermaid-"]` に以下の制約が設定されている:
  - `max-height: 490px`（スライド高さの約68%。タイトル・フッター領域を確保）
  - `min-width: 60%`（小さい図でもスライド幅の60%以上に拡大）
  - `object-fit: contain`（アスペクト比を維持してフィット）

#### AI モード Stage 2 での図の最適化

エージェントは以下の観点で描画済み Mermaid 図をレビュー・調整する:

1. **サイズ調整**: 自動設定された `w:` ヒントがスライドレイアウトに対して適切かを確認する。図がスライド下端にはみ出す場合は `w:` を小さくするか、キャプションを削減する
2. **縦長フローチャートの横方向変換**: Stage 1 が「縦長 Mermaid フローチャート」を警告した場合、以下の手順でレイアウトを最適化する:
   - `.slides.md` 内の元 Mermaid コードブロックは Stage 1 で画像に置換済みのため、**キャッシュ先の `.mmd` ファイルを直接修正**し `mmdc` で再描画する
   - `graph TD` / `flowchart TD`（縦方向）→ `graph LR` / `flowchart LR`（横方向）に変更し、16:9 スライドに適した横長レイアウトにする
   - 単純な方向変換で見た目が崩れる場合は、`subgraph` でフェーズ/ステップ単位にグルーピングし、横方向の中に縦のサブフローを含める構成にする
   - ノード数が多い場合（20ノード超）は、論理単位で複数の図に分割し、それぞれ別スライドに配置する
   - 再描画手順: `.mmd` ファイルを修正 → `mmdc -i <file>.mmd -o <file>.svg -e svg -b transparent -w <width> -c <config>` で再描画 → `.slides.md` の `w:` ヒントを更新
3. **巨大図の分割検討**: 100行を超える図（ER図等）は1スライドに収まらない可能性がある。以下の対応を検討する:
   - 図のみの専用スライドとし、タイトル + 図のみ配置（キャプションは次スライド）
   - 元文書のセクション構造を参考に、論理的なサブグラフ単位で分割を検討する
4. **図 + テキストバランス**: 図と解説テキストを同一スライドに配置する場合、テキストは1-2行のキャプションに留める。詳細解説は次スライドに分離する
5. **Mermaid コードブロック残存チェック**: `` ```mermaid `` が残存していないことを確認する（すべて画像参照に置換済みであること）

### 2-5b. 非 Mermaid コードブロック図

`` ```text `` や `` ```plaintext `` で記述されたテキスト図（ASCII アート、ボックス描画文字によるレイアウト図、組織図等）は、コードブロックのままスライドに含めると可読性が低い。

#### Stage 1（自動処理）

- **(v1.5+)** テキスト図（ASCII アート / ボックス描画）を含むコードブロックを自動検出し、「テキスト図の可能性」として警告を出力する。横幅70文字超の場合は追加の切れ警告も表示する
- **(v1.8+)** テキスト図の自動 Mermaid 変換を実施する。以下の2タイプを検出・変換する:

**自動変換の検出条件**:
- コードブロック内の行のうち30%以上が図表インジケータ（矢印 `→←`、ボックス描画文字 `┌┐└┘├│─`、バーチャート記号 `■█▓▒`、Y軸パターン `数字|`）を含む場合に変換対象とする

**対応するテキスト図タイプ**:

| タイプ | 検出パターン | 変換先 | レイアウト |
|--------|-------------|--------|-----------|
| 組織図（2列） | `[Group1] ... [Group2]` ヘッダー + `├─`/`└─` ツリー + `◄──►` クロスリンク | `graph LR` + subgraph×2 | 左右並び（`~~~` アライメント） |
| バーチャート | `+---+` 区切り + `■` バー + Y軸パターン | `xychart-beta` | 棒グラフ |

**組織図の変換ルール**:
- `graph LR` でメインフロー方向を左→右に設定（mmdc `direction TB` はサブグラフ内で非対応のため）
- 左列サブグラフ（SG1）: ノード宣言のみ。内部エッジなし
- 右列サブグラフ（SG2）: 階層エッジをサブグラフ内に配置
- 縦棒パターン（`─┐`, `─┤`, `─┘`）で間接的な親子関係を検出し、`parent --> child` エッジに変換
- クロスリンク（`◄──►`）は `<-.->` 点線で組織間連携を視覚的に区別
- サブグラフ間に `~~~` 不可視リンクで左右配置を強制（先頭ノード↔先頭ノード、末尾ノード↔末尾ノード）

**バーチャートの変換ルール**:
- データソースの優先順位: (1) 直前テーブルの太字数値 `**N**` → (2) ASCII アートの `■` 幅解析
- CJK 表示幅を考慮: `■` は表示幅2カラム。`unicodedata.east_asian_width()` で F/W/A を2カラム扱い
- `xychart-beta` の y-axis 上限は最大値+2 に設定

**自動変換が不発の場合**: インジケータ密度が30%未満の場合は変換せずコードブロックのまま残し、Stage 2 警告を出力する

#### AI モード Stage 2 での対応

エージェントは以下の基準で Stage 1 未変換の非 Mermaid コードブロック図を処理する:

1. **変換判定**: コードブロックの内容がダイアグラム的（矢印、ボックス、ツリー構造等を含む）かどうかを判断する
   - ダイアグラム的: 矢印（`→`, `←`, `-->`, `---`）、ボックス描画文字（`┌┐└┘├│─`、`+---+`）、ツリー構造（`└─`）を含む
   - 非ダイアグラム的: 通常のテキスト、コマンド出力、数値データ等
2. **ダイアグラム的な場合の変換**:
   - 内容を Mermaid 記法に変換できる場合は変換し、`mmdc` で描画して画像参照に置換する
   - Mermaid 変換が困難な場合は、コードブロックのフォントサイズを縮小する（`<span style="font-size:0.65em">` で囲む）か、箇条書きで内容を表現し直す
3. **非ダイアグラム的な場合**: セクション 2-5（コードブロック）のルールに従い、10行以内に収めるか要点抜粋する
4. **サイズ制約**: テキスト図をそのまま残す場合、横幅が70文字を超えるとスライド右端で切れる。必要に応じて折り返しまたは省略する

### 2-6. 画像・図の扱い

元文書に画像（`![キャプション](パス)`）が含まれている場合、スライドにも**必ず含める**。

- 元文書に埋め込まれた可視化図（グラフ、チャート等）は、スライドでも対応するセクションに含める
- 画像の相対パス（`figures/fig_xxx.svg` 等）はそのまま維持する。`prepare_slides_markdown.py` がソースMDの `figures/` を自動コピーするためパス解決は自動
- 1スライドに画像1枚を基本とし、画像の下に1〜2行のキャプション/要約を付記する
- 画像と詳細な解説を同じスライドに詰め込みすぎない。必要に応じて「図のスライド」と「解説のスライド」に分けてよい
- 元文書のドキュメントPDFに含まれる図がスライドPDFから欠落することは **許容しない**

### 2-6a. 手作り SVG 図（体制図・タイムライン等）

Mermaid 自動変換では品質が不十分な複雑図（体制図・組織図・タイムライン等）は、手作り SVG で作成しソース MD の `figures/` に配置する。

#### いつ手作り SVG を使うか

| 条件 | Mermaid 自動 | 手作り SVG |
|------|:---:|:---:|
| フローチャート・シーケンス図 | ○ | — |
| 単純な組織図（1階層） | ○ | — |
| カード型体制図（影・角丸・色階層） | — | ○ |
| ガントタイムライン + フェーズバー | — | ○ |
| 2組織間の連携矢印を含む体制図 | — | ○ |
| 情報量が多くレイアウト制御が必要な図 | — | ○ |

#### SVG デザイン仕様

project-slide テーマに準拠し、既存の手作り SVG（PartnerA/PartnerB の `figures/org-chart.svg`）と統一感を持たせる。

| 要素 | 仕様 |
|------|------|
| **viewBox** | `0 0 1060 H`（幅 1060 固定、高さ H は内容量に応じて 400〜620。スライド 16:9 に最適化） |
| **フォント** | `'BIZ UDPGothic','Noto Sans CJK JP','Noto Sans JP','Hiragino Sans',Arial,sans-serif` |
| **背景** | lt2 `#F7FAF9`（`<rect>` で全体背景。スライド背景と同化） |
| **ドロップシャドウ** | `<filter id="shadow"><feDropShadow dx="1" dy="2" stdDeviation="2" flood-color="#002020" flood-opacity="0.10"/></filter>` |
| **カードデザイン** | `rx="8"` 角丸 + `filter="url(#shadow)"` + 白背景 + ボーダー |
| **グループ背景** | accent6 `#E0F2F4` + accent5 `#B1DFE2` ボーダー + `rx="12"` |
| **重要ロール** | lt1 `#006970` 塗り + 白文字（PM 等） |
| **キーロール** | lt1 `#006970` ボーダー（太め `stroke-width="2"`）+ lt1 テキスト |
| **通常ロール** | dk2 `#D8DBDB` ボーダー + dk1 `#002020` テキスト |
| **接続線** | accent1 `#008587` + `opacity="0.6"` |
| **ラベル** | accent1 `#008587` 背景 8% 不透明 + lt1 テキスト |
| **矢印マーカー** | `<marker>` 要素で三角矢印（lt1 `#006970` 塗り） |
| **警告色** | `#E04040`（ピーク表示、デッドライン等。テーマ外だが視認性のため許容） |

#### ファイル配置

```text
<PJ>/<セクション>/figures/<図名>.svg   ← SSoT（手作り SVG）
docs/exports/.cache/slides/<セクション>/figures/<図名>.svg  ← 自動コピー
```

- ソース MD から `![キャプション](figures/<図名>.svg)` で相対参照する
- `prepare_slides_markdown.py` が `figures/` をキャッシュにコピーするためパス解決は自動
- Mermaid 描画済み SVG（`figures/mermaid_N.svg`）とソース `figures/` は **マージコピー**（`dirs_exist_ok=True`）で共存する

#### ソース MD の構成ルール（大きな図の場合）

手作り SVG は高さ 400px 超になることが多く、同一スライドにテーブルや長文を同居させるとはみ出す。以下のルールで H2 セクション分割する:

```markdown
## 体制概要          ← 図専用スライド（H2 単独）

![体制図](figures/org-chart.svg)

---

## 体制サマリ        ← テーブル/解説専用スライド（H2 単独）

| 項目 | 数値 |
|------|------|
| 総ロール数 | 13 |
| ピーク人数 | 10名 |
```

- `prepare_slides_markdown.py` の fast モードは **H2 境界でスライド分割** する。`---` だけでは H2 セクション内部に吸収されるため、**H2 レベルで明確に分離する**こと
- 図のみのスライドには H3 小見出しやキャプション（1行以内）を付けてよいが、テーブルは別スライドに移す
- SVG の viewBox 高さが 480px 以下で、付随テーブルが 5 行以内の場合は同一 H2 に同居してもよい

#### Stage 2 チェック項目（手作り SVG）

- [ ] ソース MD の `figures/` に SVG ファイルが存在するか
- [ ] キャッシュの `figures/` に Mermaid SVG と手作り SVG の両方が揃っているか
- [ ] SVG を含むスライドで図がはみ出していないか（viewBox 高さ + タイトル行 ≤ スライド高さ）
- [ ] 大きな図（viewBox 高さ > 480px）が H2 単独スライドになっているか

### 2-7. 更新履歴・改訂履歴の除外

成果物は社外向けであり、文書のバージョン管理履歴は不要であるため、以下のセクションは**スライドに含めない**:

- 「更新履歴」「改訂履歴」セクション（H1/H2 レベル）
- ドキュメントバージョン情報テーブル（バージョン／日付／変更内容）

> **自動除外**: `prepare_slides_markdown.py` (v1.3+) がスライド分割時に「更新履歴」「改訂履歴」セクションを自動除外する。AI モード Stage 2 でも、これらが残存していないことを確認すること。

### 2-8. 内部参照の除去

成果物は社外向けであるため、以下を含めてはならない:
- リポジトリ内のファイルパス（例: `9_Decision_Records/DR-0001-xxx.md`）
- ディレクトリ構造（例: `8_Modeling/01_前処理data/`）
- 内部ファイル名（例: `df_new_customer_credit.csv`）

成果物を参照する場合は、成果物名（「別紙 Q&A回答書」「企業リスト」等）やセクション番号で表現する。
ファイルパス列を含む表は、列自体を除去するか「備考」欄に概要を記載する。

> **自動警告**: `prepare_slides_markdown.py` (v1.4+) が Stage 1 処理時に内部参照パターンを検出し、警告を出力する。AI モード Stage 2 では、この警告を参照して除去を実施すること。

#### 検出・置換パターン一覧

| パターン | 正規表現 | 例 | 置換方法 |
|----------|---------|---|---------|
| `.md` 拡張子 | `\b[\w-]+\.md\b` | `IDM-201-機能要件定義.md` | 拡張子を除去 |
| `.csv` / `.xlsx` 拡張子 | `\b[\w-]+\.csv\b` | `IDM-002-WBS.csv` | 拡張子を除去 |
| 内部ディレクトリパス | `\b\d_\w+/` | `0_Project/`, `9_Decision_Records/` | 除去。成果物名で参照 |
| MD ファイルへのリンク | `\[text\]\(path\.md\)` | `[定義書](IDM-201.md)` | リンク除去しテキストのみ残す |
| DR 番号 | `DR-\d{3,4}` | `DR-0005`, `DR-051` | 文書タイトルに置換 |
| 内部データファイル名 | `df_\w+`, `log_\w+` | `df_new_customer_credit.csv` | 説明的な日本語名に置換 |

#### 置換の判断基準

1. **ファイル拡張子**（`.md`, `.csv`, `.xlsx`）→ **常に除去**
2. **ディレクトリプレフィクス**（`N_xxx/`）→ **常に除去**
3. **DR 番号** →
   - `export-config.json` に `output_name` がある → その名称に置換（例: DR-0005 → 「計画・スコープ変更説明書」）
   - ない場合 → DR 本文のタイトルから推定
4. **内部ファイル名**（`df_xxx`, `log_xxx` 等）→ 内容を示す日本語名に置換
5. **Markdown リンク記法** `[text](path)` → リンク除去しテキストのみ残す

### 2-9. レイアウトクラスの使い分け

- **表紙（1枚目）**: `<!-- _class: lead -->` + `<!-- _paginate: false -->` を必ず付与。**テーブル（メタデータ表含む）は表紙に配置しない**。メタ情報は2枚目に移動
- **大章の区切り**: `<!-- _class: section -->` でセクションヘッダーを挿入する
  - ただし、全体スライド数を不要に増やさないため、章数が少ない（5章以下）場合のみ使用する
- **2カラムで比較・並列表示が適切な内容**: `<!-- _class: two-col -->` を使用する
- **通常のコンテンツ**: クラス指定なし（TITLE_AND_BODY レイアウト）

### 2-10. コンテンツのはみ出し防止

スライド下端で文字や図が切れることを防ぐため、以下のルールを遵守する:

- **情報量の上限目安**: 1スライドあたり、本文テキスト + 表 + リストの合計で **スライド高さの90%以内** に収める
- **表の行数制限**: 1スライドに表を配置する場合、表の行数は **12行以内**（ヘッダー含む）。超える場合は要約または分割する
- **フォントサイズ調整**: コンテンツが溢れる場合、以下の順で調整する:
  1. コンテンツの要約・圧縮（最優先）
  2. 表のフォントサイズ縮小（`<span style="font-size:0.7em">` で囲む）
  3. 画像サイズの縮小（`![alt w:500](path)` → `![alt w:400](path)`）
  4. スライド分割（最終手段）
- **画像の最大高さ**: 単独画像は `max-height: 70%`（スライド高さの70%）を目安とし、キャプション・タイトルの領域を確保する
- **CSS安全策**: テーマに `overflow: hidden` が設定されているため、溢れたコンテンツは切断される。**AI モードでは必ず目視相当のチェックを行い、切断が発生しない構成にすること**

> **チェック方法**: Stage 2 レビューでは、各スライドの行数・表行数・画像サイズを確認し、上記基準を超えている場合は調整すること。

## 3. フォーマットルール

### 3-1. Marp Front Matter

生成する `.slides.md` の先頭には以下を固定挿入する:

```yaml
---
marp: true
theme: project-slide
size: 16:9
paginate: true
footer: TikTok Affiliate Automation
---
```

### 3-2. スライド区切り

- スライド間は `---` で区切る
- 各スライドの先頭行は必ず `# タイトル` とする

### 3-3. 水平線の除外

- 元文書内の水平線（`---`, `***`, `___`）はスライド区切りと衝突するため除外する

## 4. AI モードワークフロー

### 4-1. 3段階処理

1. **Stage 1（機械生成）**: `prepare_slides_markdown.py --mode ai` が `fast` モードと同じルールベース変換で初版を生成し、キャッシュに保存する
2. **Stage 2（エージェントレビュー）**: エージェントがキャッシュ済み `.slides.md` を読み込み、本ガイドのルールに従って構成を最適化する
3. **Stage 3（はみ出しゼロ検証）**: Stage 2 完了後、全スライドを対象にコンテンツがスライド下端を超えていないかを最終検証する。はみ出しが1箇所でもあれば修正し、再検証する

### 4-2. Stage 2 チェックリスト

エージェントは以下の観点でレビュー・修正を行う:

- [ ] 各スライドにタイトル（`# ...`）があるか
- [ ] タイトルだけのスライドが存在しないか（表紙・セクション区切り除く）
- [ ] 1つの章が3スライド以上に分割されていないか
- [ ] 「続きN」ラベルが残っていないか（自然なタイトルに置換）
- [ ] 表が不要に分割されていないか
- [ ] 箇条書きが1スライド8項目を大幅に超えていないか
- [ ] 全体ページ数が元文書の章数の 1.5倍 以内か
- [ ] Front Matter が正しいか（marp, theme, size, paginate, footer）
- [ ] **表紙に `<!-- _class: lead -->` と `<!-- _paginate: false -->` が付与されているか**
- [ ] **表紙スライド（`lead`）にテーブルが含まれていないか**（表紙はタイトルとサブタイトルのみ）
- [ ] **レイアウトクラスが適切に使い分けられているか**（セクション 2-8 参照）
- [ ] 内部参照（ファイルパス・ディレクトリ構造・内部ファイル名）が含まれていないか（セクション 2-8 のパターン一覧を参照。**Stage 1 の警告出力も確認すること**）
- [ ] **ファイル拡張子（`.md`, `.csv`, `.xlsx`）がスライド本文中に残っていないか**
- [ ] **Markdown リンク記法 `[text](path.md)` で内部パスを参照していないか**
- [ ] **更新履歴・改訂履歴セクションが含まれていないか**（セクション 2-7 参照）
- [ ] 元文書に含まれる画像がすべてスライドにも含まれているか（欠落禁止）
- [ ] 画像の相対パスが正しいか（`figures/` ディレクトリの存在を確認）
- [ ] Mermaid コードブロック（`` ```mermaid ``）が残存していないか（すべて画像参照に置換済みであること）
- [ ] **Mermaid 図の `w:` ヒントがスライドレイアウトに適切か**（図 + タイトル + キャプションがはみ出さないこと）
- [ ] **巨大な Mermaid 図（100行超の元コード）が適切に配置されているか**（専用スライドまたは分割）
- [ ] **Stage 1 の「縦長 Mermaid フローチャート」警告に対処したか**（§2-5a の TD→LR 変換手順に従い、`.mmd` ファイルを修正・再描画）
- [ ] **非 Mermaid コードブロック図（`` ```text `` 等）がダイアグラム的な場合、可読性のある形式に変換されているか**
- [ ] **Stage 1 の「テキスト図の可能性」警告に対処したか**（Mermaid 変換・フォント縮小・箇条書き化のいずれかを適用）
- [ ] **テキスト図の横幅が70文字を超えてスライド右端で切れていないか**
- [ ] **手作り SVG（`figures/` 内の非 mermaid SVG）がキャッシュに正しくコピーされているか**
- [ ] **大きな手作り SVG（viewBox 高さ > 480px）が H2 単独スライドになっているか**

### 4-3. Stage 3 はみ出しゼロ検証

Stage 2 の構成最適化が完了した後、**最終ゲート**として以下を全スライドで検証する。
1箇所でも違反があれば修正し、再度本チェックを通すまでPDF化しない。

- [ ] **表の行数が12行以内か**（ヘッダー含む）。超過 → 要約 or 分割
- [ ] **画像 + キャプション + タイトルがスライド高さに収まるか**。超過 → 画像縮小 or 分割
- [ ] **箇条書き + 本文の総量がスライド高さの90%以内か**。超過 → 要約 or 分割
- [ ] **CSS `overflow: hidden` による切断が発生しない構成になっているか**

> **重要**: `overflow: hidden` はあくまで表示崩れ防止の安全策であり、コンテンツが切断されたまま納品することは**絶対に許容しない**。Stage 3 で切断を検出した場合は必ず修正すること。

### 4-4. 修正方法

- エージェントはキャッシュ済み `.slides.md` を直接編集する
- 元の SSoT Markdown は変更しない
- 修正後のファイルはそのまま `export_slides_pdf.py` への入力として使用する

## 5. 品質基準

| 指標 | 基準 |
|------|------|
| スライド数 | 元文書の H1/H2 章数の 1〜1.5 倍 |
| タイトルのみスライド | 0（表紙・セクション区切り除く） |
| 「続きN」ラベル | 0 |
| 表の分割 | 元表ごとに1スライド以内 |
| 箇条書き項目 | 1スライドあたり 8 以内 |
| 画像の欠落 | 0（元文書の画像はすべてスライドにも含める） |
| Mermaid コード残存 | 0（すべて描画済み画像に置換） |
| 更新履歴セクション | 0（自動除外 + AI確認） |
| コンテンツはみ出し | 0（全スライドがスライド下端内に収まること） |
| 図のサイズ | 全図がスライド内に収まり、下端・右端で切断なし |
| 図の日本語 | Mermaid 図内の日本語テキストが文字化けなく表示 |
| 内部参照 | DR番号・ファイル拡張子・ディレクトリパスがスライド本文にゼロ |
| 表紙クラス | `lead` 付与済み |
| 手作り SVG | ソース `figures/` とキャッシュ `figures/` の両方に存在。大きな図は H2 単独スライド |
| デザイン準拠 | PPTX カラーパレット・レイアウト対応 |
