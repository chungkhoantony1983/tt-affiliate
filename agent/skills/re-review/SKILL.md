作業やコンテキストが大きくなることによって精度が落ちないように、作業を分解して、適切な順番で実行して下さい。

### Coverage Manifest 引き継ぎ（re-review 固有）

re-review では前回のレビュー状態を引き継ぎ、効率的な再レビューを実施する:

1. **Prior Manifest**: 前回の CoverageManifest（ReviewUnit 構造）を引き継ぎ、Phase 0 で impact/delta モードで再生成する。変更のあった Unit のみ再スキャンし、未変更 Unit は前回結果を再利用
2. **Deferred Register**: 前回「day2 対応」等で見送った DeferredItem を引き継ぐ。Cross-Impact Check（CR-013）により、依存先ファイル（`deferral_basis.depends_on`）に変更があった項目は自動 re-raise される
3. **Cross-Impact 自動検出**: 新規 findings のターゲットファイルが DeferredItem の依存先と一致する場合、見送り理由の前提が崩壊したとみなし、自動的に再浮上させる

**前回レビューログとの対応**:
- レビューログの `## Deferred Register` セクションから DeferredItem を復元
- 前回の findings ID 体系を継続（`F-{counter}` の連番を引き継ぎ）

共通初期化プロトコル完了後、`workflow_step.py` でフェーズ順序を強制管理する。init は teams.yaml + learned-rules.yaml をプログラム的に読み込み、`config_context`（ペルソナ・制約・ルール・フェーズ担当者）を返す。各フェーズの実行時はこの config_context に従うこと。

| ステップ | コマンド | タイミング |

| advance | `python .claude/scripts/workflow_step.py advance --state {state_file} --result {result.json}` | 下記マッピングの各境界 |

**SKILL.md → teams.yaml フェーズマッピング**（advance は各グループ完了後に1回）:

| teams.yaml Phase | SKILL.md Phase（このグループ完了後に advance） |
|-----------------|----------------------------------------------|
| requirements | requirements |

- `{state_file}`: init の出力 `state_file` キーから取得。セッション終了時に削除（C-IMP071-4）

### Phase: requirements

**学習ルールの確認**: `.claude/references/rules/learned-rules.yaml` を読み込み、Global および該当ドメイン（`code-review` — Step 0 b' で確定）の constraints/processes を確認する（ファイル不存在時はスキップ）。レビュー時はこれらの制約への違反も検出対象とする。

**enforcement 分類の活用**: `hook_strict` ルールは PreToolUse hook で自動ブロックされるため、hook 通過を確認すれば十分。`hook_fuzzy` ルールは hook では検出困難なためレビュー時に手動で確認すること。`prose_gate` ルールはレビュー観点として使用する。レビュー報告時に、関連する `hook_fuzzy` / `prose_gate` ルールの検証結果を明記すること。

Step 0 b' で解決されたドメインのペルソナ（`persona_refs × phase_affinity` で各フェーズに自動割当）がディスカッションしてレビュー方針を決定します。
以下に記載した別のレビュアーによるコメントを踏まえ、ソースコードとDR、テスト、e2eテストシナリオを改めて多角的にレビュー・テストの実行をして下さい。
もしあなたが過去にレビュー・テストを実施して、別のレビュアーによるコメントと比較して考慮漏れがあったのであれば、該当の漏れだけでなく、漏れた根本原因を考え、横展開してレビューをし直し、変更範囲についての全範囲とその影響する範囲全てにおいてレビュー漏れがないことを確認して下さい。
もしも反論があるのであれば、反論を論拠や証左とともに明示して下さい。
あなたの再レビュー結果について、修正が必要か否かを断定できるまでディスカッションして下さい。
修正が必要な範囲については、具体的な根拠と修正範囲、修正方針を明示して下さい。
後から追加で指摘が出てくることのないように、特定の箇所をレビューして懸念が出れば、横展開で同様の課題がないかをチェックしてから、別の箇所のレビュー、ディスカッションを繰り返し、変更範囲についての全範囲とその影響する範囲全てにおいてレビュー漏れがないことを確認して下さい。
特にリグレッションやデグレードは意図した仕様変更でない限りは禁止です。
報告は日本語でお願いします。

### Phase: architecture

### 合意形成プロトコル（必須）

単一エージェントの独断判定を排し、複数視点の相互批評で品質を担保する。
詳細は `role-workflow-guide.md` の「合意形成プロトコル」セクションを参照。

**Tier 1 実装（Claude Task × 3）:**

1. **Phase 1 — 独立分析**: 3つの Task エージェントを並列起動し、異なる観点で独立にレビューする
   - エージェントA: 構造・整合性（アーキテクチャ、データフロー、型安全性）
   - エージェントB: 表記・用語統一（命名規則、コメント、ドキュメント整合性）
   - エージェントC: 業務正確性・コード一致（仕様との一致、エッジケース、リグレッション）

### Phase: implement

2. **Phase 2 — 相互レビュー**: 3エージェントの出力を交換し、各出力を他2つが再レビューする Task を実行
   - A の指摘 → B, C が観点漏れ・誤りを指摘
   - B の指摘 → A, C が同様にレビュー
   - C の指摘 → A, B が同様にレビュー
3. **Phase 3 — 反論解決**: 不一致点を抽出し、反論が出なくなるまで繰り返す

### Phase: review

4. **Phase 4 — 収束判定**: 全エージェントの指摘が完全一致するまで Phase 2 に戻る

**禁止事項:**
- 単一エージェントのレビュー結果を無条件で採用しない
- 最初のラウンドで「PASS」判定を出さない（最低1回の相互レビュー必須）
- 収束前にレビュー完了としない

### 0ベースレビュー実施基準（必須）

以下を厳守すること:

1. **対象範囲**: 変更セット全体（全変更ファイル全文）。修正箇所のみの狭域レビューは禁止
2. **前回結果非参照**: 0ベース（ゼロから全体を再検証）。前回レビューの指摘リストを見ながらのチェックリスト式レビューは禁止
3. **横展開（GP-002 / RV-006）**: 特定箇所で懸念が出たら、grep/検索で同パターンが他箇所にないか全数確認してから次の箇所へ。防御的チェック（isinstance, .get(), None判定等）が追加されている場合は、同一パターン全箇所に横展開されていることを検証する
4. **全ペルソナ参加**: 5ペルソナ全員がレビューに参加し、各自の指摘を明示

### 前回レビューログの自動読み込み

1. **検索順序**（4+1パターン自動判定に対応）:
   a. **(A) パス埋め込み型 / (B) PJサブフォルダ型**: Step 0 でパス解決済み → `{REVIEW_LOG_PATH}/review-*.md` を直接検索
   b. **(C) サブフォルダ検出型**: `{DR_PATH}` 直下でPR番号含みサブフォルダ（例: `pr-6730-*`）を検索し、`{サブフォルダ}/.review-logs/review-*.md` を検索
   c. **(C') 混在**: PR番号サブフォルダ内（`{DR_PATH}/pr-{PR番号}-*/.review-logs/review-*.md`）と `{REVIEW_LOG_PATH}/review-*.md` の両方を検索し、見つかったものをすべて候補とする
   d. **(D) フラット / フォールバック**: `{REVIEW_LOG_PATH}/review-*.md` を検索
2. 「未到達」状態の最新ログを選択
3. **ログが見つかった場合**:
   - 前回の指摘一覧・ステータスを読み込み
   - 各指摘IDの解決状況を対象ファイルと照合して検証
   - `$ARGUMENTS` に追加情報があればそれも考慮
4. **ログが見つからない場合**:
   - `$ARGUMENTS` のみで従来通り再レビューを実施
   - 完了後にログファイルを新規作成し Round 1 として記録（配置先は上記検索順序と同じロジックで決定）

### Phase: report

### レビューログへの追記

再レビュー完了後、ログファイルに新ラウンドを追記:

1. `## Round N — /re-review (<日付>)` を追加（N = 前回 + 1）
2. 前回指摘の検証テーブル（ID・前回判定・今回判定・根拠）を記入
3. 新規指摘があれば追加（IDは前回最大ID + 1 から連番）
4. ペルソナ合議サマリ・ステータスを記入
5. コンセンサスセクションを更新
6. **ユーザーへの報告時にレビューログファイルのパスを明記すること**

ログ形式の詳細は `.claude/skills/review/references/review-log-format.md` を参照。

### 次のアクション案内

再レビュー結果の報告後、以下を案内すること：

- **修正が必要な場合**: `/fix` で課題修正を依頼
- **修正不要の場合**: `/push` でコミット・プッシュ
- **レビュー履歴をPRにコメントする場合**: `/comment` でレビュー履歴の解説を投稿
- **厳格なマルチモデルレビューが必要な場合**: `/ai-review` でコンセンサスレビュー（3モデル×7ペルソナ、収束基準あり）
