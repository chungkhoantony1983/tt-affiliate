# レビューログ形式仕様

## 目的

- レビュー結果を構造化ファイルに記録し、セッション/ツール間（Claude↔Codex、Claude Chat 1↔Chat 2）で引き継ぐ
- 「prompt is too long」対策: 前回会話全文ではなく構造化サマリを読み込む
- DRは引き続き正式な決定記録。レビューログは作業用の中間成果物

## 配置ルール

全リポで **PR番号サブフォルダ方式** に統一。`{DR_PATH}` = `docs/decision-records/{PR番号}/`、`{REVIEW_LOG_PATH}` = `{DR_PATH}/review-logs/`。

PR番号は PR 作成時に確定する（`gh pr view` で取得）。wpm のみ `{PJフォルダ}/docs/decision-records/{PR番号}/` がプレフィックスになる。

```
<repo>/{REVIEW_LOG_PATH}/review-<YYYY-MM-DD>-<pj-slug>.md
  → 例: project/docs/decision-records/6730/review-logs/review-2026-02-20-module-a.md
  → 例: ea/docs/decision-records/120/review-logs/review-2026-02-20-ea.md
  → 例: workspace-pm/ProjectA/docs/decision-records/42/review-logs/review-2026-02-20-project-a.md
```

### 共通ルール

- `pj-slug`: PJディレクトリ名の小文字（`project-a`, `partner-b`, `77`, `partner-d`）
- 日付はレビューサイクル開始日
- 1サイクルにつき1ファイル（`/re-review` で同一ファイルにラウンド追記）
- **コミット対象外**: レビューログはworktree内に一時保存し、コミットには含めない。worktree削除時に自動消滅する（明示的にコミットを指示された場合を除く）

## ログフォーマット

```markdown
# レビューログ: <PJ名> (<YYYY-MM-DD>)

対象: <対象の説明（ブランチ名、変更概要など）>
対象DR: DR-XXXX, DR-YYYY
対象ファイル: IDM-001, IDM-004, ...

---

## Round 1 — /review (YYYY-MM-DD)

### 指摘一覧

| ID | 重大度 | カテゴリ | 対象 | 指摘内容 | 判定 |
|----|--------|---------|------|---------|------|
| F-001 | クリティカル | データ整合性 | IDM-004:L141 | 人日不一致 | 修正必要 |
| F-002 | メジャー | トレーサビリティ | IDM-001:§成果物 | 参照先欠落 | 修正必要 |

### ペルソナ合議サマリ

- **合意**: F-001, F-002 は修正必要
- **横展開**: データ整合性を全テーブルで確認済み

### ステータス

未解決: F-001, F-002
解決済み: （なし）

---

## Round 2 — /re-review (YYYY-MM-DD)

### 前回指摘の検証

| ID | 前回判定 | 今回判定 | 根拠 |
|----|---------|---------|------|
| F-001 | 修正必要 | 解決 | DR-0026で基準人日ベースに再構成 |
| F-002 | 修正必要 | 解決 | IDM-001 §成果物一覧を更新 |

### 新規指摘

（なし / テーブル）

### ペルソナ合議サマリ

- **合意**: F-001, F-002 解決確認

### ステータス

未解決: （なし）
解決済み: F-001, F-002

---

## コンセンサス

**判定**: 全指摘解決（コンセンサス達成） / 未到達
**次アクション**: DR作成 → /export / /fix → /re-review Round N
```

## フォーマット詳細

### 指摘ID

- `F-001` から連番で付与
- ラウンドをまたいで一意（Round 2 で新規指摘が出たら前回最大ID + 1 から開始）

### 重大度

| レベル | 意味 |
|--------|------|
| クリティカル | データ不整合・仕様欠陥・セキュリティ問題 |
| メジャー | トレーサビリティ欠如・テスト不足・設計上の問題 |
| 中 | 要件の曖昧さ・可読性の問題 |
| マイナー | 表記揺れ・フォーマット |
| 軽微 | 誤字・体裁 |

### 判定ステータス

- `修正必要` — 修正が必要と合議された
- `解決` — 修正が確認された
- `対応不要（合意済み）` — 合議の結果、修正不要と判断された

## コンセンサス判定基準

以下をすべて満たしたとき「コンセンサス達成」:

1. 未解決の指摘がゼロ
2. 最新ラウンドで新規のクリティカル/メジャー指摘がゼロ
3. 5ペルソナ全員が解決に合意
4. 横展開チェックで類似問題なし

## 自動検出ロジック（/re-review 用）

1. **PR指定ありの場合**: Step 0 でパス解決済み → `{REVIEW_LOG_PATH}/review-*.md` を直接検索
2. 「コンセンサス」セクションが「未到達」のファイルを選択
3. 複数あれば最新日付のファイルを使用
4. 見つからない場合は `$ARGUMENTS` のみで従来通り実施
