作業やコンテキストが大きくなることによって精度が落ちないように、作業を分解して、適切な順番で実行して下さい。

### アーキテクチャ: Coverage Manifest ベース ReviewUnit 分割

大規模 PR では変更ファイルを **ReviewUnit** に分割し、Unit 単位でサブエージェントにレビューを委任する:

1. **Unit 分割**: 変更ファイル + 関連ファイル（テスト・import 先）をデータフロー的にグループ化し、トークン上限（~20K tokens/Unit）に収まるよう分割
2. **サブエージェント × N Units**: 各 Unit を独立した Task エージェントに委任（Phase 1: 並列スキャン）
3. **Coverage Gate**: 全 Unit がスキャン済みであることを確認してから合議フェーズに進む（CR-012）
4. **Probe Scan**: findings=0 の高複雑度 Unit に対してターゲット質問を実行し、早期収束を防止（CR-014）
5. **横展開検証**: Unit 間の依存関係（CrossUnitLink）を追跡し、影響先 Unit も検証する

**PR サイズ別の動作**:
- **Small**（<100行, ≤3ファイル）: Unit 分割なし、従来通り 3 エージェント並列
- **Medium/Large**: Unit 分割 + Coverage Gate 適用

### Phase: requirements

**学習ルールの確認**: `.claude/references/rules/learned-rules.yaml` を読み込み、Global および該当ドメイン（`code-review` — Step 0 b' で確定）の constraints/processes を確認する（ファイル不存在時はスキップ）。レビュー時はこれらの制約への違反も検出対象とする。

**enforcement 分類の活用**: `hook_strict` ルールは PreToolUse hook で自動ブロックされるため、hook 通過を確認すれば十分。`hook_fuzzy` ルールは hook では検出困難なためレビュー時に手動で確認すること。`prose_gate` ルールはレビュー観点として使用する。レビュー報告時に、関連する `hook_fuzzy` / `prose_gate` ルールの検証結果を明記すること。

Step 0 b' で解決されたドメインのペルソナ（`persona_refs × phase_affinity` で各フェーズに自動割当）がディスカッションしてレビュー方針を決定します。
ソースコードを参照してDRの理解を深め、DRに記載の仕様が要件、構造、骨格、表層の欠陥を抱えていないか、より良い要件、構造、骨格、表層はないか、e2eのテストシナリオがドキュメントに記載されているか、シナリオの網羅性があるかををレビューして下さい。
加えて、ソースコードとDR、テスト、e2eテストシナリオを多角的にレビューし、テストの実行をして下さい。
レビュー結果について、修正が必要か否かを断定できるまでディスカッションして下さい。
修正が必要な範囲については、具体的な根拠と修正範囲、修正方針を明示して下さい。
後から追加で指摘が出てくることのないように、特定の箇所をレビューして懸念が出れば、横展開で同様の課題がないかをチェックしてから、別の箇所のレビュー、ディスカッションを繰り返し、変更範囲についての全範囲とその影響する範囲全てにおいてレビュー漏れがないことを確認して下さい。
特にリグレッションやデグレードは意図した仕様変更でない限りは禁止です。
変更範囲に関連する既存コードの技術負債（コード重複、非推奨パターン、ベストプラクティス逸脱等）を検出した場合は、「本PRの差分ではない」として棄却せず、day2 指摘として記録し `/issues` で Issue 化して下さい。レビューは技術負債を可視化・起票する貴重な機会であり、見つけ次第 Issue 化しないと返済タイミングを逸します。
報告は日本語でお願いします。

### Phase: architecture

### 合意形成プロトコル（必須）

単一エージェントの独断判定を排し、複数視点の相互批評で品質を担保する。
詳細は `role-workflow-guide.md` の「合意形成プロトコル」セクションを参照。

**Tier 1 実装（Claude Task × 3）:**

1. **Phase 1 — 独立分析**: 3つの Task エージェントを並列起動し、異なる観点で独立にレビューする
   - エージェントA: 構造・整合性（アーキテクチャ、データフロー、型安全性）
   - エージェントB: 表記・用語統一（命名規則、コメント、ドキュメント整合性）
   - エージェントC: 業務正確性・コード一致（仕様との一致、エッジケース、リグレッション）
### Phase: implement

2. **Phase 2 — 相互レビュー**: 3エージェントの出力を交換し、各出力を他2つが再レビューする Task を実行
   - A の指摘 → B, C が観点漏れ・誤りを指摘
   - B の指摘 → A, C が同様にレビュー
   - C の指摘 → A, B が同様にレビュー
3. **Phase 3 — 反論解決**: 不一致点を抽出し、反論が出なくなるまで繰り返す

### Phase: review

4. **Phase 4 — 収束判定**: 全エージェントの指摘が完全一致するまで Phase 2 に戻る
5. **インライン修正後の0ベースレビュー再実施（必須）**: レビュー中に指摘を発見しインラインで修正した場合（DRテキスト修正・コード修正の軽重を問わず）、**修正後に Phase 1 から0ベースレビューを再実施してから結果を報告すること**。「軽微な修正だから再レビュー不要」という判断は禁止

**禁止事項:**
- 単一エージェントのレビュー結果を無条件で採用しない
- 最初のラウンドで「PASS」判定を出さない（最低1回の相互レビュー必須）
- 収束前にレビュー完了としない

### 並行変更の暗黙的依存チェック（CR-008 — CI pr-overlap.yml の補完）

Phase 1 の独立分析と並行して、CI のファイルレベル検出では捕捉できない**暗黙的依存**を検出する。

1. `gh pr list --state open --json number,headRefName --jq '.[] | .number'` で同一リポの全オープン PR を取得
2. 各オープン PR の変更ファイル一覧を `gh pr diff {number} --name-only` で収集
3. **CI 結果確認**: pr-overlap.yml がファイルレベルの重複を既に検出済みか確認。検出済みなら AI レビューは暗黙的依存に集中する
4. **暗黙的依存の分析**:
   - module include / mixin の副作用（変更ファイル外のクラスに影響する include/extend/prepend）
   - メソッド委譲チェーン（変更メソッドの呼び出し元・呼び出し先が他 PR で変更されていないか）
   - DB スキーマ依存（マイグレーション順序衝突、カラム参照の整合性）
   - DR 前提条件の崩壊（他 PR の変更が本 PR の DR 前提を無効化していないか）
5. 暗黙的依存を検出した場合は **Critical 指摘** として報告する
6. 検出結果をレビューログに記録する
7. 参照: `development-guide.md` §2「並行変更の検出と保護」

### 0ベースレビュー実施基準（必須）

以下を厳守すること:

1. **対象範囲**: 変更セット全体（全変更ファイル全文）。修正箇所のみの狭域レビューは禁止
2. **前回結果非参照**: 0ベース（ゼロから全体を再検証）。前回レビューの指摘リストを見ながらのチェックリスト式レビューは禁止
3. **横展開（GP-002 / RV-006）**: 特定箇所で懸念が出たら、grep/検索で同パターンが他箇所にないか全数確認してから次の箇所へ。防御的チェック（isinstance, .get(), None判定等）が追加されている場合は、同一パターン全箇所に横展開されていることを検証する
4. **全ペルソナ参加**: 5ペルソナ全員がレビューに参加し、各自の指摘を明示

### Phase: report

### レビューログの記録

レビュー完了後、結果を構造化ログに記録する:

1. **配置先の決定**（4+1パターン自動判定。Step 0 で解決済みのパス変数を優先使用）:
   - **(A) パス埋め込み型** / **(B) PJサブフォルダ型**: Step 0 で `{REVIEW_LOG_PATH}` が具体パスに解決済み → そのまま直接使用
   - **(C) サブフォルダ検出型**: `{DR_PATH}` 直下でPR番号またはブランチ名を含むサブフォルダ（例: `pr-6730-*`）を検索し、見つかれば `{DR_PATH}/{サブフォルダ}/.review-logs/` に配置。見つからない（新規PR）場合は `{DR_PATH}/pr-{PR番号}-{kebab-title}/.review-logs/` を作成
   - **(C') 混在**: `{DR_PATH}` 直下に PR番号サブフォルダと直置き DR が共存する場合 → 新規レビューログはサブフォルダ側に配置（`{DR_PATH}/pr-{PR番号}-{kebab-title}/.review-logs/`）。既存ログ検索は両方を対象とする
   - **(D) フラット** / **フォールバック**: `{REVIEW_LOG_PATH}` に配置
2. 配置先ディレクトリを作成（なければ）
3. `review-<今日の日付>-<PJスラッグ>.md` を新規作成
4. ヘッダー（対象・DR・ファイル一覧）を記入
5. `## Round 1 — /review (<日付>)` として指摘一覧テーブル・合議サマリ・ステータスを記入
6. IDは `F-001` から連番。重大度: クリティカル / メジャー / 中 / マイナー / 軽微
7. コンセンサスセクションに判定を記入
8. **ユーザーへの報告時にレビューログファイルのパスを明記すること**

ログ形式の詳細は `.claude/skills/review/references/review-log-format.md` を参照。

### 次のアクション案内

レビュー結果の報告後、以下を案内すること：

- **修正が必要な場合**: `/fix` で課題修正を依頼

- **レビュー履歴をPRにコメントする場合**: `/comment` でレビュー履歴の解説を投稿
- **厳格なマルチモデルレビューが必要な場合**: `/ai-review` でコンセンサスレビュー（3モデル×7ペルソナ、収束基準あり）
