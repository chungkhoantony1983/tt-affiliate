# エクスポート標準（export-standard）

> **正本宣言**: このファイルはエクスポート運用ルールの唯一の正本。`/export` スキル（SKILL.md）はこの標準を参照する手順書である。

## 1. 適用範囲

- `/export` スキルによる成果物生成（スライド PDF / Excel / ガントチャート）
- Tier 1 スキル（Claude Code + Codex、全リポで使用可能）

## 2. 成果物配置ルール（§4-7 準拠）

- **出力先**: `docs/exports/{deliverable-name}/`（成果物単位のサブフォルダ）
- 出力先ディレクトリは `os.makedirs(exist_ok=True)` で自動作成すること（手動作成不要）
- SSoT（変換元 MD / CSV）と同じフォルダへの出力は**禁止**
- `docs/exports/` ディレクトリが `.gitignore` で除外されていないことを確認

### 2-0. 成果物ライフサイクル（§4-7 新設）

`docs/exports/` は成果物（deliverable）単位で管理する。各成果物は設計定義（SSoT）→ 中間生成物（作業ログ）→ バージョン付きエクスポート（外部共有資料）のライフサイクルを持つ。

```
docs/exports/{deliverable-name}/
├── brief.yaml                       ← SSoT: 成果物の設計定義（スライドの場合）
├── reference/                       ← SSoT: 元資料（PDF原本等、オンデマンド作成）
├── drafts/                          ← 作業ログ: 中間生成物（HTML, 下書きExcel等）
│   ├── slide_01.html
│   └── ...
├── {name}_{ver}_{date}.pdf          ← 外部共有資料: バージョン付きエクスポート
└── {name}_{ver}_{date}.xlsx         ← 外部共有資料: バージョン付きエクスポート
```

| 層 | 区分 | 例 | Git 管理 |
|:---|:---|:---|:---:|
| **設計定義** | SSoT | `brief.yaml` | Yes |
| **中間生成物** | 作業ログ | `drafts/slide_*.html`, `drafts/*.xlsx` | 場合による |
| **バージョン付きエクスポート** | 外部共有資料 | `investor-deck_v1.0_20260305.pdf` | Yes |

### 2-1. 生成スクリプト配置ルール

PJ固有の生成スクリプト（Excel変換・ガントチャート・カスタムレポート等）は以下に配置し、再実行可能にすること:

- **配置先**: `<PJルート>/scripts/gen_{種別}.py`（例: `gen_wbs.py`, `gen_issues_excel.py`）
- スクリプトは引数なし or 最小限の引数で再実行可能であること
- 出力先パスはスクリプト内でデフォルト値を持つこと（`docs/exports/` 配下）

## 3. 命名ルール（§4-7 準拠）

エクスポートファイルの命名規則: `{document-name}_{version}_{date}.{ext}`

| 要素 | 形式 | 例 |
|:---|:---|:---|
| `{document-name}` | ケバブケース、成果物名と一致 | `investor-deck`, `api-spec`, `credit-risk-report` |
| `{version}` | `v{major}.{minor}` | `v1.0`, `v2.1` |
| `{date}` | `YYYYMMDD` | `20260305` |
| `{ext}` | ファイル形式 | `pdf`, `pptx`, `xlsx` |

例:
- `investor-deck_v1.0_20260305.pdf`
- `credit-risk-report_v2.1_20260228.pdf`
- `issues-summary_v1.0_20260301.xlsx`

- DRを含む場合は `output_name` で受領者向けタイトルを指定すること（内部DR番号をファイル名に含めない）

## 4. 出力形式

| 種別 | ツール | 備考 |
|------|--------|------|
| Markdown → スライド PDF | `export_slides_pdf.py`（Marp） | 全Markdown文書は `slide_source_files` に配置。`markdown_files` は使用しない |
| CSV → Excel | `export_csv_to_excel.py`（openpyxl） | — |
| WBS CSV → ガントチャート Excel | `export_wbs_gantt.py`（openpyxl） | — |
| 一括実行 | `export_project_deliverables.py --config {PJ}/scripts/export-config.json --date {YYYY-MM-DD}` | 可能な限りこちらを使用 |

- **PDF出力はスライド形式のみ**: 通常文書PDFは作成しない
- **更新履歴は自動除外**: スライド変換時に「更新履歴」「改訂履歴」セクションを除外（顧客向け成果物には不要）
- **DRの変更管理書運用は継続**: DRをまとめた変更管理書の別途PDF化は変更しない

## 5. セルフチェック要件

### 5-1. 一般チェック

- 文字化けがないこと
- 画像・Mermaid図が正しく描画されていること
- **表レイアウト検証**: テーブルの右側カラムが見切れていないこと。特に多カラムテーブル（5列以上）は代表ページをPDF上で目視確認すること。CSSの `table-layout` は `auto`（コンテンツベース幅）を使用し、`fixed`（均等幅）は使用しないこと
- ページ構成が意図通りであること
- **はみ出しなし**: スライド下端での文字・図の切断がないこと（`slide-composition-guide.md` のコンテンツ制限ルールに従う）
- **内部参照除去**: スライド本文に `DR-XXXX`、`.md`、ディレクトリパスが残っていないこと
- **システムパス除去**: `/Users/`、`/tmp/`、`/home/`、`.claude/`、`C:\` 等のローカルファイルパスが成果物に含まれていないこと
- **図表レンダリング完了**: ` ```mermaid ` ブロックや未変換のマークアップが生テキストとして残っていないこと（mermaid → SVG 変換が完了していること）
- **UI要素混入防止**: PDF/PPTX出力にスクロールバー、ツールバー、カーソル等のUI要素が写り込んでいないこと（スクリーンショットベースの出力時に特に注意）

### 5-2. 課題管理の関心事分離

> **原則1**: 課題管理CSVは外部共有課題専用とし、内部タスクは起票しない

> **原則2**: レポート・仕様書のスライドMDに課題テーブルを埋め込まない。課題管理はCSVがSSoTであり、インライン埋め込みは二重管理となるため禁止。「別紙: 課題管理表」「課題一覧」等を発見した場合は、SSoT（元MD）とキャッシュ（.slides.md）の両方から削除し、参照先を課題CSV側に記載する

> **原則3**: `公開区分` 列は防御的措置として維持する

**エクスポート時の必須チェック**:

1. **公開区分フィルタ設定**: 課題管理CSV（ファイル名に「課題」「issues」を含む）の `export-config.json` エントリに以下が設定されていること:
   - `filter_column`: フィルタ対象列（例: `公開区分`）
   - `filter_value`: 公開対象の値（例: `外部`）
   - `drop_columns`: 成果物から除外する列（`filter_column` を含めること）
   - **未設定の場合はエクスポートを中断**し、設定を追加してから再実行

2. **課題テーブル埋め込み禁止チェック**: 原則2に違反するインライン課題テーブルがないこと

3. **内部課題混入チェック**: `公開区分=内部` の課題IDがスライドMD本文に残っていないこと。発見した場合はSSoT（元MD）とキャッシュ（.slides.md）の両方から削除

### 5-3. 集計サマリ要件

ステータス管理を含むデータ（課題管理CSV、WBS等）をエクスポートする場合、**サマリ行を必ず表示**すること:

- **課題管理CSV**: 総数 / Open件数 / Closed件数 / Critical+High の Open件数
- **WBS**: カテゴリ別人日合計 / 総人日 / 総人月
- **体制表**: ロール別合計人数

サマリが欠落しているエクスポートは完了としない。

## 6. コミット要件

- 成果物は作業ブランチに**必ずコミット**すること（漏れ厳禁・スキップ禁止）
- `git status` で `docs/exports/` 配下に untracked / modified ファイルがないことを確認
- コミットメッセージに対象PJ・提出日・ファイル一覧を含める
- コミット完了後に再度 `git status` で `docs/exports/` 配下の未追跡ファイルがゼロであることを確認
- 未追跡ファイルが残っている場合はエクスポート完了としない

## 7. AIスライドレビュー（Stage 2 → Stage 3）

`slide_source_files` に `mode: ai` が指定されている場合に適用。

### Stage 2: 構成最適化

- ファイルごとに **Task サブエージェントに委譲**（コンテキスト溢れ防止）
- `.slides.md` が 80行未満ならメインエージェントで直接実施可
- `slide-composition-guide.md` セクション 4 のチェックリスト全項目を検証・修正

### Stage 3: はみ出しゼロ検証

- 全スライドを走査:
  - 表の行数: 12行以内
  - 画像高さ: 70%以内
  - 総量: 90%以内
- **1箇所でもはみ出しがあれば修正し、再検証を通すまでPDF化しない**
