# DR作業計画テンプレート

## 目的

- `/task` スキルおよび `/fix`, `/spec-update` スキルのPhase 1（Planning）で起票するDRの標準フォーマット
- 意思決定記録（従来のDR）に加え、作業計画・完了基準（DoD）・実績を一体管理する
- ユーザー承認ゲートの判断材料として機能する
- **テスト駆動**: 実装前に期待結果（コード・ドキュメント両方）を定義し、デシジョンテーブルで入力パターンごとの振る舞いを明確化する

## 配置ルール

- `<PJ>/{DR_PATH}/DR-XXXX-<kebab-case-title>.md`
- 番号は各PJ内の連番（既存DRの最大番号 + 1）
- 従来の意思決定DRと同一ディレクトリに格納（区別は内容で判断）

## DR不変ルール

- **mainにマージされたDRは改変不可**（内容の修正・追記・削除すべて禁止）
- 既存DRの対象範囲を変更する必要がある場合は、**新規DRを起票**し `supersedes` で旧DRの該当箇所を参照する
- 制約の解除・変更も新規DRで行う（旧DR側は一切変更しない）

## `/push` スキルとの同期要件

`/push` 実行時、DRには以下が記録されている必要がある（レビューログからの転写確認項目）:

1. **根本原因**: コンテキスト → 課題セクションに記載
2. **修正方針**: 決定セクション + 構造変更セクションに記載
3. **検証基準**: 完了基準（DoD）セクション → デシジョンテーブル + E2E検証チェックリストに記載

## 使い分け

- **意思決定DR**: 設計判断・技術選定など、決定事項の記録が主目的のとき
- **作業計画DR**: `/task`, `/fix`, `/spec-update` で作業を委任するとき（計画→承認→実行フロー）
- 1つのDRが両方の性質を持つこともある（例: 技術選定を伴う実装タスク）

## pushスキルとの対応

| pushスキルの確認項目 | DRテンプレートの対応セクション |
|-------------------|--------------------------|
| 根本原因 | コンテキスト → 課題（根本原因） |
| 修正方針 | 決定（修正方針） + 構造変更 |
| 検証基準 | 完了基準（DoD） → デシジョンテーブル + E2E検証チェックリスト |
| 制約転記 | 制約事項 → CONSTRAINTS.mdに転記（新規追加 / superseded制約の削除） |
